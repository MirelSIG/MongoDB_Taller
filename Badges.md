1. Que es MongoDB y cuales son sus principales características?
MongoDB es una base de datos NoSQL orientada a documentos que almacena datos en formato JSON-like. Sus principales características incluyen:

- Esquema flexible: No requiere un esquema fijo, lo que permite almacenar datos de manera más natural y adaptarse a cambios en los requisitos sin necesidad de modificar un esquema rígido.
- Escalabilidad horizontal: Permite distribuir los datos en múltiples servidores o nodos a través de su arquitectura de sharding, lo que facilita el manejo de grandes volúmenes de datos y tráfico.

1. Por que es importante para un dev identificar las cargas de trabajo y optimizar el esquema de DDBB para ellas en MongoDB?
Porque MongoDB es un sistema de base de datos NoSQL y no tiene un esquema fijo, lo que permite a los desarrolladores diseñar su base de datos de manera flexible. Sin embargo, esto también significa que es crucial identificar las cargas de trabajo específicas para optimizar el esquema de la base de datos. Al entender las cargas de trabajo, los desarrolladores pueden diseñar un esquema que se adapte mejor a las consultas y operaciones que se realizarán con mayor frecuencia, lo que puede mejorar significativamente el rendimiento y la eficiencia de la base de datos.
2. Cual es la diferencia entre un esquema de base de datos relacional y uno NoSQL como MongoDB?
La principal diferencia entre un esquema de base de datos relacional y uno NoSQL como MongoDB es que las bases de datos relacionales utilizan un esquema fijo y estructurado, donde los datos se organizan en tablas con filas y columnas, y las relaciones entre los datos se establecen mediante claves foráneas. En cambio, MongoDB es una base de datos NoSQL que utiliza un esquema flexible y dinámico, donde los datos se almacenan en documentos JSON-like dentro de colecciones. Esto permite a los desarrolladores almacenar datos de manera más natural y adaptarse a cambios en los requisitos sin necesidad de modificar un esquema rígido.
3. Que es un índice en MongoDB y como puede mejorar el rendimiento de las consultas?
Un índice en MongoDB es una estructura de datos que mejora la velocidad de las operaciones de consulta en una colección. Al crear un índice en uno o más campos de una colección, MongoDB puede buscar y recuperar documentos de manera más eficiente, ya que el índice permite acceder a los datos sin tener que escanear toda la colección. Esto puede mejorar significativamente el rendimiento de las consultas, especialmente en colecciones grandes, al reducir el tiempo de búsqueda y aumentar la velocidad de respuesta.
4. Cual es la importancia de la escalabilidad en MongoDB y como se logra?
La escalabilidad es una característica fundamental de MongoDB, ya que permite a las aplicaciones manejar grandes volúmenes de datos y tráfico sin comprometer el rendimiento. MongoDB logra la escalabilidad a través de su arquitectura de sharding, que distribuye los datos en múltiples servidores o nodos. Esto permite a las aplicaciones escalar horizontalmente, agregando más nodos al clúster para manejar el aumento de la carga de trabajo. Además, MongoDB ofrece características como la replicación y la tolerancia a fallos, lo que garantiza la disponibilidad y la resiliencia de los datos incluso en situaciones de alta demanda o fallos del servidor. En resumen, la escalabilidad en MongoDB es crucial para garantizar que las aplicaciones puedan crecer y manejar el aumento de datos y tráfico de manera eficiente.
5. Que es un documento en MongoDB y como se relaciona con las colecciones?
Un documento en MongoDB es una unidad de datos que se almacena en formato JSON-like. Cada documento es una estructura de datos que puede contener campos y valores, y puede anidar otros documentos o arrays. Los documentos se almacenan en colecciones, que son agrupaciones de documentos relacionados. Las colecciones en MongoDB son equivalentes a las tablas en las bases de datos relacionales, pero a diferencia de las tablas, las colecciones no requieren un esquema fijo, lo que permite a los desarrolladores almacenar datos de manera más flexible y adaptarse a cambios en los requisitos sin necesidad de modificar un esquema rígido. En resumen, un documento es la unidad básica de datos en MongoDB, y las colecciones son agrupaciones de documentos relacionados que permiten organizar y gestionar los datos de manera eficiente.
6. Cual es la diferencia entre una consulta de lectura y una de escritura en MongoDB?
Una consulta de lectura en MongoDB se refiere a la operación de recuperar datos de la base de datos, mientras que una consulta de escritura se refiere a la operación de modificar o agregar datos en la base de datos. Las consultas de lectura pueden incluir operaciones como find() para buscar documentos, mientras que las consultas de escritura pueden incluir operaciones como insertOne() para agregar un nuevo documento o updateOne() para modificar un documento existente. En resumen, las consultas de lectura se centran en recuperar datos, mientras que las consultas de escritura se centran en modificar o agregar datos en la base de datos.
7. Que es la replicación en MongoDB y como puede mejorar la disponibilidad de los datos?
La replicación en MongoDB es el proceso de copiar datos de un servidor principal (primary) a uno o más servidores secundarios (secondary) para garantizar la disponibilidad y la resiliencia de los datos. La replicación permite que los datos estén disponibles incluso en caso de fallos del servidor principal, ya que los servidores secundarios pueden asumir el rol de servidor principal en caso de una falla. Esto mejora la disponibilidad de los datos, ya que los usuarios pueden seguir accediendo a la base de datos incluso si el servidor principal no está disponible. Además, la replicación también puede mejorar el rendimiento de las consultas de lectura, ya que los servidores secundarios pueden manejar las consultas de lectura mientras el servidor principal se encarga de las operaciones de escritura. En resumen, la replicación en MongoDB es una característica clave que mejora la disponibilidad y la resiliencia de los datos, garantizando que las aplicaciones puedan seguir funcionando incluso en situaciones de alta demanda o fallos del servidor.
